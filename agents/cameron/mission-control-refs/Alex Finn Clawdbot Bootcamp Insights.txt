📚 Teaching Section Summary
What is Mission Control?


Mission Control is a home for custom tooling in OpenClaw. It's a locally-hosted Next.js dashboard where you build tools specific to your workflows. OpenClaw has skills and memories out of the box, but Mission Control is where you add custom functionality.
Essential Tools Everyone Needs:


1.Calendar— Track scheduled tasks, keep OpenClaw accountable for proactive work
2.Memory Viewer— See every memory file and edit, search through memories
3. Docs Viewer— View all documents OpenClaw creates without digging through folders
4. Team Tool— See all your agents like an org chart
5.Mission Statement— North Star displayed at top of Mission Control
Mission Statement Advice:
* Your mission statement is your North Star for everything you and OpenClaw do
* Alex's: "An autonomous organization of AI agents that does work for me and produces value 24/7"
* If you don't know yours, reverse prompt: "Based on what you know about me, what could be a mission statement we could use?"
* Having it burned into OpenClaw's memory means when you say "do 5 tasks tonight," it aligns to your goals
Mission Control Philosophy:
* Should be personalized to YOU — not a copy of Alex's
* As you work, when something feels unoptimized, just say "hey OpenClaw, build that for me"
* A gift store owner's MC was purely dashboards/analytics for his brick & mortar store
* Content creators → content pipelines. E-commerce → sourcing/shipping dashboards
* Reverse prompt: "Based on our workflows, what should be the first tools we build in our mission control?"
Demo: Feedback-to-Feature Pipeline
* Built a system where Creator Buddy users submit feedback → goes to Mission Control dashboard
* One-click button triggers AI to build the feature → creates a branch in GitHub
* You review and ship. Built the entire system in 25 minutes with OpenClaw.

1. Reverse Prompting is Everything — Don't just tell your AI what to do. Ask it questions: "What should we build? How should we approach this? What's the best architecture?"

5. Security Awareness — Anyone with access to your OpenClaw has admin access to everything on that machine. Plan accordingly.
6. OpenClaw for Planning, Claude Code/Codex for Building — Use OpenClaw's context advantage for strategy, dedicated tools for deep coding.
7. Convex for Real-Time — If your Mission Control doesn't update live, implement Convex (free, built-in real-time subscriptions).
8. Mission Statement = North Star — Burn it into your OpenClaw's memory. Every task, every decision should align to it.
9. Watch for Scams — Always verify sender domains on emails claiming to be from tools you use.

📚 TEACHING SECTION SUMMARY
WHAT IS THE SOFTWARE FACTORY?
• A closed-loop system where you give a direction to OpenClaw, and a team of agents autonomously builds, ships, and continuously improves software
• Alex's example: A multiplayer game that gets built, released, and constantly improved by AI agents 24/7 — new features, weapons, skins, game mechanics — all without human intervention
• Players can submit feedback, and agents read it and build what's requested
• This concept isn't just for software — it can be a content factory, sports reporting factory, or any automated production pipeline
MULTI-AGENT ARCHITECTURE (THE TEAM)
• Henry — Main orchestrator agent on Mac Studio 1, powered by Anthropic Opus 4.6. The "Chief of Staff" you communicate with directly.
• Charlie — Dedicated coder on Mac Studio 2, powered by Quen 3.5 (local model). Codes 24/7 at $0 cost.
• Ralph — Engineering manager on Mac Studio 2, powered by ChatGPT OAuth (Codex 5.3). Checks on Charlie every 20 minutes to ensure he's on the right path (a "Ralph loop").
• Violet — Researcher on Mac Mini, powered by Minimax 2.5 (local on Mac Studio 1). Researches trends, Product Hunt, TrustMRR 24/7 and feeds ideas to Ralph.
Key principle: Only communicate with ONE main agent (Henry). Henry orchestrates everyone else. Don't coordinate with 20 agents directly.
SUB-AGENTS VS. SEPARATE OPENCLAWS
• Sub-agents are NOT real agents — they're just your main agent "putting on different hats." No separate memories, skills, or instructions.
• Sub-agents do allow parallel work, but lack true multi-agent benefits.
• Separate OpenClaws are full independent agents with their own memories, skills, workspaces, and availability (24/7).
• Alex is phasing out sub-agents in favor of separate OpenClaws.
• Each OpenClaw needs its own AI model/account.
• Can run multiple OpenClaws on the same device — just watch memory usage.
HOW TO SET UP MULTIPLE OPENCLAWS
• Use OpenClaw profiles: openclaw new profile Ralph creates a new instance
• Run with: openclaw profile Ralph
• Or just tell your OpenClaw: "Set up a new OpenClaw" and it handles everything
• Communication between OpenClaws on different devices: Use SSH
• Shared workspace: Henry set up shared memory files so all agents can see what each other is working on, while each also has private workspaces
MODEL RECOMMENDATIONS
1. Best brain: Anthropic Opus 4.6 — smartest, warmest, most human-like
2. Great backup: ChatGPT OAuth — very good, more robotic personality, way more usage/tokens
3. Budget options: Minimax, Kimi K, Quen (Chinese models) — good for worker agents
4. Worker agents (coding/research) don't need the smartest models — save genius-level for orchestration
5. ChatGPT $100/month plan (new) expected to be great value
6. ChatGPT $250/month = "an AI employee for $250/month" — incredible value
7. Keep using Anthropic OAuth until they kick you off — it's that good
FACTORY MISSION CONTROL SCREENS
• Team Screen — Shows all agents, their roles, devices, models. Provides structured context for the orchestrator.
• Factory Screen — 2D pixel art animation showing agents working, Kanban board tracking tasks across stages, agent statuses and current work.
• Calendar Screen — Shows all scheduled cron jobs so you can verify proactive work is actually scheduled.
• Philosophy: Build tools you enjoy using. Have fun with it. The pixel art factory is fun and that's reason enough.
TWO PATHS TO MAKING MONEY WITH AI
1. SaaS Factory (Shotgun Approach) — AI researches trending products, builds your own versions, pump out apps, see what gets customers. Works best with an audience/platform.
2. Niche Sliver (Sniper Approach) — Build one app for one very specific use case and persona (e.g., "CRM for Korean grocery stores"). OpenAI/Anthropic will never target $2-3M markets. That's your opportunity.
LOCAL MODELS EXPLAINED
• Cloud models: Your request goes to OpenAI/Anthropic servers, computation happens there, answer returns
• Local models: Computation happens entirely on your own device. No internet needed, completely private, $0 cost (just electricity)
• Chinese companies (Quen, Kimi, Minimax) open-source their models — downloadable and runnable locally
• Downsides: Chinese models aren't as good as American ones; need powerful hardware (32GB+ memory recommended)
• Security: Local models are just math equations — no network activity, no data theft. Would be caught immediately if they tried.
• New Quen 3.5 model: ~20GB, runs on 32GB machines, roughly Sonnet 4.5 quality
KEY TIPS FROM THE SESSION
1. Build a Mission Control — Custom dashboard for your workflows. Team view, factory view, calendar view.
2. One orchestrator, multiple workers — Talk to one agent. Let it manage the rest.
3. Use local models for 24/7 workers — Free, private, always available.
4. Use smart cloud models for decision-making — Opus for brain, cheaper models for muscle.
5. Everything is fixable — When agents screw up, ask what went wrong and how to prevent it. Self-improving system.
6. Build your platform — Social media/distribution is critical for solo ventures in the AI age.
7. Wait for M5 Ultra — Mac Studio wait times extended to 3 months, suggesting imminent M5 announcement. Massive leap in inference speed.

Key Tips/Guidance
💡 Tip: Self-Auditing Your Claude Bot


Write a prompt asking your Claude to create a self-audit, then use Opus or Codex (in VS Code or Cursor) to review every file in its memory/heartbeat/soul. This helps find conflicts, assumptions, and things you've written that are preventing the bot from achieving its goals. Alex found massive conflicts this way—things he told it days apart that contradicted each other.
💡 Tip: Build Transparency Tools for Auditing


Create an activity feed so you can see every activity your Claude bot does—this lets you audit what it's doing. Also build a calendar so you can see what's scheduled. If your bot says it'll do something proactively but it's not on the calendar, you can question it: "What's going on here?"
💡 Tip: Have Claude Build Its Own Tooling


When building tooling or functionality for your Claude bot, have the bot do it itself because it has the best context around itself and knows what to plug into. Only have Claude bot build something out otherwise if you're on the go and want a prototype ready when you get home.
💡 Tip: Set Up an Org Structure for Multi-Step Workflows


Configure your Claude bot with different "employees" in the org structure (PRD builder, project manager, lead developer). When giving commands, it will know to leverage the right role—having the PRD builder create the document first, then hand it to the developer. This is a more structured approach than straight-up vibe coding.
